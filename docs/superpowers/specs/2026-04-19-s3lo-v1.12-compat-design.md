# s3lo-operator: s3lo v1.12 Compatibility & Feature Parity

**Date:** 2026-04-19  
**Status:** Approved

## Objective

Upgrade s3lo-operator to be fully compatible with s3lo v1.12.3, fixing a broken build caused by the `pkg/s3` → `pkg/storage` API rename, adding multi-arch Image Index support (operator v1.2.0 milestone), switching to presigned URL blob serving (performance), and adding cosign-based image signature verification at serve time (operator v2.0.0 milestone).

---

## 1. API Migration — `pkg/s3` → `pkg/storage`

### Problem

s3lo removed `pkg/s3` in v1.2.0. The operator imports it in two places; upgrading the dep breaks the build.

### Changes

| File | Change |
|------|--------|
| `go.mod` | Bump `github.com/OuFinx/s3lo` from `v1.1.0` to `v1.12.3`; run `go mod tidy` |
| `cmd/s3lo-proxy/main.go` | `s3client "github.com/OuFinx/s3lo/pkg/s3"` → `storage "github.com/OuFinx/s3lo/pkg/storage"`; `s3client.NewClient` → `storage.NewS3Client` |
| `pkg/proxy/handlers.go` | Same import swap; type `*s3client.Client` → `*storage.Client`; remove local `isNotFound()` helper, use `storage.IsNotFound()` instead; remove direct `aws-sdk-go-v2/service/s3` and `s3types` imports from proxy package |
| `pkg/proxy/server.go` | Import swap; type ref update |

`storage.Client.ClientForBucket(ctx, bucket)` retains the same signature and region-detection behavior — no call-site logic changes.

## 2. Multi-arch Image Index Support (operator v1.2.0)

### Problem

The manifest handler hardcodes `Content-Type: application/vnd.oci.image.manifest.v1+json`. s3lo v1.3.0+ stores OCI Image Indexes at the same manifest path with `mediaType: application/vnd.oci.image.index.v1+json`. containerd negotiates by content type; serving an Index with the wrong type breaks multi-arch pulls.

Additionally, after containerd receives an Image Index it requests individual platform manifests by digest. The current `DigestCache` only stores manifests fetched by tag — it has no fallback for index children.

Finally, HEAD requests to the manifest endpoint return a body, which is incorrect HTTP behavior.

### Changes to `HandleManifest`

**Content-Type detection:**
Add `mediaTypeFromManifest(data []byte) string` that unmarshals only the `mediaType` field from the JSON and returns it; falls back to `application/vnd.oci.image.manifest.v1+json` if absent. Use this for `Content-Type` and to inform cache behavior.

**HEAD handling:**
After writing response headers, check `r.Method == http.MethodHead` and return without writing the body.

**Digest-based manifest lookup:**
1. Check `DigestCache` (existing — fast path for tag-fetched manifests).
2. New fallback: attempt `GetObject` on `blobs/sha256/<encoded>` in the requested bucket. s3lo v1.3.0+ stores index child manifests there. If found, serve with correct content type and return.
3. If both miss: return `MANIFEST_UNKNOWN 404`.

**v1.0.0 layout fallback:** unchanged.

## 3. Presigned URL Blob Serving

### Problem

`HandleBlob` currently streams S3 object bytes through the proxy via `io.Copy`. For multi-hundred-MB image layers this consumes proxy CPU, memory, and network bandwidth on every pull from every node.

### Design

Replace streaming with a presigned redirect:

1. Derive the S3 key: `blobs/sha256/<hex>` (v1.1.0) or cached path (v1.0.0 fallback).
2. If HEAD: call `HeadObject` to confirm existence; return 200 with `Docker-Content-Digest` header.
3. If GET: call `storage.Client.PresignGetObject(ctx, bucket, key, ttl)` and respond `307 Found` with `Location: <presigned-url>`.

No image layer data passes through the proxy process. containerd follows the redirect natively.

**Configuration:**
- `S3LO_PRESIGN_TTL` env var — presign TTL duration (default: `1h`). Must exceed the maximum time any single layer download could take.

**v1.0.0 fallback:** for blobs still keyed by per-tag path (legacy layout), resolve the path from `DigestCache` as before, then presign that path.

## 4. Signature Verification at Serve Time (operator v2.0.0)

### Problem

Kubernetes nodes pull whatever image is specified in a Pod spec. Without verification, a compromised or tampered image pushed to S3 will be served silently.

### Signature storage format (from s3lo v1.9.0)

s3lo stores signatures at:
```
manifests/<image>/<tag>/signatures/<keyid-slug>.json
```
where `keyid-slug` is a 16-char hex prefix of `SHA-256(DER-encoded public key)`.

The JSON record:
```json
{
  "schemaVersion": 1,
  "digest": "sha256:<manifest-digest>",
  "keyRef": "<original-key-ref>",
  "keyID": "<slug>",
  "algorithm": "ECDSA_SHA_256",
  "signature": "<base64>",
  "payload": "<base64 of 'sha256:<digest>\n'>",
  "signedAt": "2026-04-19T12:00:00Z"
}
```

### New file: `pkg/proxy/verify.go`

```
Verifier struct
  verifier  cosignsig.Verifier  (cosign interface)
  slug      string              (16-char hex, computed once at startup)
  cache     sync.Map            (digest → bool, verified results)

NewVerifier(ctx, keyRef string) (*Verifier, error)
  Calls cosignsig.VerifierForKeyRef — supports PEM files, awskms://, gcpkms://, azurekms://
  Falls back to cosignsig.SignerVerifierFromKeyRef for .key private-key files
  Pre-computes slug from public key

(*Verifier).Check(ctx, s3Client *storage.Client, bucket, image, ref string, manifestData []byte) error
  1. Compute sha256 of manifestData → digest string
  2. Check cache: if digest already verified, return nil immediately
  3. Fetch signatures/<slug>.json via s3Client.GetObject(ctx, bucket, sigKey) — no raw SDK needed
     - NotFound → return ErrNotSigned
  4. Unmarshal record; check rec.Digest == digest → return ErrDigestMismatch if not
  5. Decode rec.Signature + rec.Payload; call verifier.VerifySignature
     - Failure → return ErrInvalidSignature
  6. On success: store digest in cache, return nil
```

### Integration in `HandleManifest`

After fetching manifest bytes by tag (not for digest-based requests):
- If `Verifier` is set: call `v.Check(...)`.
- On any error: write HTTP 403 with OCI body `{"errors":[{"code":"DENIED","message":"image not permitted: invalid or missing signature"}]}` and return.
- On nil: proceed normally.

Digest-based requests skip verification — they are index children pulled after the index was already verified by tag.

### Configuration

| Env var | Default | Description |
|---------|---------|-------------|
| `S3LO_VERIFY_SIGNATURES` | `false` | Set to `true` to enable verification |
| `S3LO_KEY_REF` | `""` | cosign key ref: file path or `awskms://arn:...` or `gcpkms://...` |

### Helm chart (`deploy/helm/s3lo-operator/values.yaml`)

```yaml
signing:
  enabled: false
  keyRef: ""        # awskms:// ARN, gcpkms:// URL, or mounted file path
  keySecret:
    name: ""        # name of existing K8s Secret containing the public key
    key: "key.pub"  # key within the secret
```

When `signing.enabled=true`:
- Sets env vars `S3LO_VERIFY_SIGNATURES=true` and `S3LO_KEY_REF=<keyRef>`.
- If `signing.keySecret.name` is non-empty: mounts the secret as a volume at `/etc/s3lo/signing/` so the file path in `keyRef` resolves correctly.
- If `keyRef` is a KMS ARN: no volume mount needed; IAM handles authentication.
